# mohan-dotfiles: repo guide

Personal machine setup as a Nix flake (standalone Home Manager, non-NixOS). The source of truth for every config is this repo; the live files under `$HOME` are read-only Nix store symlinks, refreshed by `home-manager switch --flake ~/REPO/mohan-dotfiles --impure`.

## Layout

* `flake.nix` pins nixpkgs and home-manager (release-25.05), reads the account name from `$USER` at eval time (hence `--impure` everywhere below), and defines five checks: building the home configuration, the Claude and Copilot hook regression suites, the Copilot status line unit tests, and the `setup.sh` lint + doctor tests.
* `nix/` holds one module per concern (`packages.nix`, `zsh.nix`, `git.nix`, `agents.nix`, `claude.nix`, `copilot.nix`, `nvim.nix`, `pi.nix`), all imported by `nix/home.nix`.
* `claude/`, `copilot/`, `nvim/`, `pi/` hold the actual config content the modules deploy. `agents/` (linked to `~/.agents` by `nix/agents.nix`) is the common layer: tool-agnostic utility scripts and templates shared by Claude Code, Copilot CLI, and Pi, e.g. `agents/skills/arch/` (HTML template + Node.js injection script behind both `/arch` ports). It holds only real shared artifacts, never prose, with one exception: `agents/AGENTS.md` is the single authored global system prompt for all three tools. Claude Code supports `@path` imports, so `claude/CLAUDE.md` is just the one-line pointer `@~/.agents/AGENTS.md`. Copilot CLI and Pi have no import directive, so `nix/copilot.nix` and `nix/pi.nix` instead generate `~/.copilot/copilot-instructions.md` and `~/.pi/agent/AGENTS.md` at build time via `builtins.readFile ../agents/AGENTS.md`; neither `copilot/copilot-instructions.md` nor a `pi/`-side copy exists in the repo, since the store-rendered copy is the only copy. `agents/skills/` also reaches Pi for free: Pi natively auto-discovers `~/.agents/skills/*/SKILL.md` (the Agent Skills standard), so `nix/agents.nix`'s existing `~/.agents` symlink is the only wiring needed — unlike `claude.nix` and `copilot.nix`, which each link `agents/skills` into a tool-specific path. Each tool has its own status line script (`claude/statusline-usage.py`, `copilot/statusline-usage.py`); they are tool-specific (different data models: Claude reads its live usage API, Copilot sums this month's AI credits from `~/.copilot/session-state/*/events.jsonl`), so they are not shared and not ports of each other. Pi has no status line script yet.
* `setup.sh` is the single entry point: no-arg run does new-machine setup and applies config changes (wraps `home-manager switch`), `setup.sh doctor` audits that every managed file is still served from the Nix store, `setup.sh upgrade` bumps flake inputs then applies. Its lint + doctor tests run as the `setup-script` flake check.
* `docker/` is an optional fully-isolated alternative to the host installs above: a Dockerfile with one stage per tool (`claude`, `copilot`, `pi`), a `docker-compose.yml` mounting the caller's `$PWD` (via `$REPO_PATH`) to `/workspace`, and per-tool named volumes for `$HOME/.claude`, `$HOME/.copilot`, `$HOME/.pi` so auth/session state persists in Docker's own storage, never on the host filesystem. Config (hooks, skills, AGENTS.md, the pi sandbox extension) is baked into `/opt/agent-config/` at build time, then `entrypoint.sh` re-syncs it into the volume on every container start — never a one-time `COPY` straight into the tool's config dir, since a named volume only inherits an image directory's content on its very first mount; re-syncing on every start means a rebuilt image always takes effect on the next run, without touching the credential/session files already in the volume. The `claude` stage's baked `settings.json` has its scaffold-toolkit hook entries stripped via `jq` (see the Dockerfile), and the `copilot` stage's baked hooks directory omits `scaffold-toolkit.json` outright: those hooks call into `~/.claude/scaffold` / `~/.copilot/scaffold`, live out-of-store symlinks to a host checkout (`~/REPO/scaffold-toolkit/...`) that isn't mounted into the container. `zsh/agent-containers.zsh` (`agent-box`/`claude-box`/`copilot-box`/`pi-box`) wraps `docker compose run` with `$REPO_PATH` set to the caller's cwd. Since Docker is now the isolation boundary for these containers, the inner bwrap-based sandboxes (`claude/settings.json`'s `sandbox.enabled`, `pi/agent/extensions/sandbox`) aren't specially reconfigured for nesting — both degrade gracefully (a warning, not a hard failure) if bubblewrap can't start inside the container, per their own documented behavior.

## Rules for changes

* A config edit in this repo does nothing until `home-manager switch --impure` runs; always mention that when finishing a config change.
* `nix flake check --impure` must pass before committing; it is also the CI gate (`.github/workflows/ci.yml`).
* Hook scripts must keep runtime state under `${XDG_STATE_HOME:-$HOME/.local/state}/claude-hooks` (see `claude/hooks/lib/common.sh`) or `.../copilot-hooks` (see `copilot/hooks/scripts/lib/common.sh`); `~/.claude/hooks` and `~/.copilot/hooks` are read-only store links, so writing next to the scripts fails.
* New hooks need an entry in the matching `test-hook.sh` (`claude/hooks/` or `copilot/hooks/`: HOOK_INFO + default payload) and, when behavior is testable, a selftest case. Both selftests run sandboxed in CI via the flake checks.
* `copilot/hooks` is a shell port of `claude/hooks` for Copilot CLI (camelCase payloads, JSON decisions on stdout instead of exit-2 blocks; see `copilot/hooks/mohan-hooks.json`); `pi/agent/extensions/hooks` is the TypeScript port, a Pi extension (`pi.on(...)` event handlers) that shells out to the same `claude/hooks/*.sh` scripts wherever Pi's tool-input shape matches Claude's closely enough to translate (loop breaker, digest generation, the post-edit type-check/build/sonar-lite chain, session cleanup). Two pieces are reimplemented natively instead of shelled out, because the shapes don't map cleanly: the edit no-op guard (Pi's `edit` tool takes `{path, edits: [{oldText, newText}]}`, an array, not Claude's single `{file_path, old_string, new_string}` — verify against `dist/core/tools/edit.d.ts` in the installed `@earendil-works/pi-coding-agent` package before assuming otherwise) and goal-capture/goal-check (Pi's `turn_end` event hands over the assistant's message directly, so there's no transcript file to parse — but Pi also has no confirmed way to block a turn from ending the way Claude's Stop hook does, so the goal-check side only warns via `ctx.ui.notify` rather than gating). Keep the ports in sync when one changes; unlike `copilot/hooks` and `claude/hooks`, `pi/agent/extensions/hooks` has no `test-hook.sh`-equivalent selftest wired into the flake checks yet.
* `pi/agent/extensions/sandbox` is Pi's own `examples/extensions/sandbox` (vendored verbatim from the installed `@earendil-works/pi-coding-agent` package, not third-party), wrapping `@anthropic-ai/sandbox-runtime` — the same bubblewrap-based tech `claude/settings.json`'s `sandbox.enabled` uses. `nix/pi.nix` copies it into a writable `~/.pi/agent/extensions/sandbox/` (not a store symlink, since it needs its own `npm install`) rather than linking it read-only like `hooks/`. Pi has no built-in OS-level sandbox of its own (confirmed in the installed package's `docs/security.md`); this extension is the only isolation layer. GitHub Copilot is a built-in `/login` subscription provider in Pi (confirmed in `docs/providers.md`) — no proxy, no `registerProvider()` extension, and no config file; it's a one-time interactive OAuth step, not something a nix activation script can do.
* The same port rule applies to the rest of the prompt config: `copilot/agents/*.agent.md` and `copilot/skills/*/SKILL.md` mirror `claude/agents/*.md`, `claude/skills/*/SKILL.md`, and `claude/commands/*.md` (adapted surfaces: Copilot has no custom slash commands, agent frontmatter uses `tools: [...]` lists, skills use YAML `name:` + `description:` frontmatter, and slash-command-equivalent skills like `/arch` and `/featurePlan` live at `claude/commands/<x>.md` while their Copilot counterparts live at `copilot/skills/<x>/SKILL.md`). A change to an agent, command, or skill on one side must land on the other. The global instructions (`agents/AGENTS.md`) are the exception, not a mirrored pair; see above.
* Before writing a duplicate file across `claude/` and `copilot/`, check whether it belongs in `agents/` instead: if it's a script, template, or other artifact with zero tool-specific content (like `agents/skills/arch/`'s template and injection script), it goes in `agents/` once and both tools reference it by its fixed `~/.agents/...` path. Prose that merely reads similarly (agent personas, skill instructions) still doesn't qualify for that pattern, since Copilot CLI's agents and skills have no import directive the way `agents/AGENTS.md` does; that keeps following the port rule above.
* Deployment exceptions, do not "fix" them: `~/.config/nvim` is an out-of-store symlink to this checkout (LazyVim writes `lazy-lock.json`), and `~/.claude/settings.json` is a writable copy (Claude Code writes permission grants into it).
* Secrets never enter this repo; machine-local values belong in `~/.zshrc.local` (sourced by `nix/zsh.nix`). On first adoption on a machine, `setup.sh`'s `migrate_pre_nix_dotfiles` folds any pre-existing `~/.zshrc`/`~/.zshenv` content into `~/.zshrc.local` before Home Manager takes the files over, so nothing is silently lost.
* Claude Code and the npm global CLIs are intentionally not pinned by Nix; they install natively via activation scripts only when missing.
