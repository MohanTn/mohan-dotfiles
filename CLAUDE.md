# mohan-dotfiles: repo guide

Personal machine setup as a Nix flake (standalone Home Manager, non-NixOS). The source of truth for every config is this repo; the live files under `$HOME` are read-only Nix store symlinks, refreshed by `home-manager switch --flake ~/REPO/mohan-dotfiles --impure`.

## Layout

* `flake.nix` pins nixpkgs and home-manager (release-25.05), reads the account name from `$USER` at eval time (hence `--impure` everywhere below), and defines five checks: building the home configuration, the Claude and Copilot hook regression suites, the Copilot status line unit tests, and the `setup.sh` lint + doctor tests.
* `nix/` holds one module per concern (`packages.nix`, `zsh.nix`, `git.nix`, `agents.nix`, `claude.nix`, `copilot.nix`, `nvim.nix`, `pi.nix`), all imported by `nix/home.nix`.
* `claude/`, `copilot/`, `nvim/`, `pi/` hold the actual config content the modules deploy. `agents/` (linked to `~/.agents` by `nix/agents.nix`) is the common layer: tool-agnostic utility scripts and templates shared by Claude Code and Copilot CLI, e.g. `agents/skills/arch/` (HTML template + Node.js injection script behind both `/arch` ports). It holds only real shared artifacts, never prose; `claude/` and `copilot/` keep their own instruction files even when the wording is similar, since there's no runtime mechanism here for one tool to include another's file. Each tool has its own status line script (`claude/statusline-usage.py`, `copilot/statusline-usage.py`); they are tool-specific (different data models: Claude reads its live usage API, Copilot sums this month's AI credits from `~/.copilot/session-state/*/events.jsonl`), so they are not shared and not ports of each other.
* `setup.sh` is the single entry point: no-arg run does new-machine setup and applies config changes (wraps `home-manager switch`), `setup.sh doctor` audits that every managed file is still served from the Nix store, `setup.sh upgrade` bumps flake inputs then applies. Its lint + doctor tests run as the `setup-script` flake check.

## Rules for changes

* A config edit in this repo does nothing until `home-manager switch --impure` runs; always mention that when finishing a config change.
* `nix flake check --impure` must pass before committing; it is also the CI gate (`.github/workflows/ci.yml`).
* Hook scripts must keep runtime state under `${XDG_STATE_HOME:-$HOME/.local/state}/claude-hooks` (see `claude/hooks/lib/common.sh`) or `.../copilot-hooks` (see `copilot/hooks/scripts/lib/common.sh`); `~/.claude/hooks` and `~/.copilot/hooks` are read-only store links, so writing next to the scripts fails.
* New hooks need an entry in the matching `test-hook.sh` (`claude/hooks/` or `copilot/hooks/`: HOOK_INFO + default payload) and, when behavior is testable, a selftest case. Both selftests run sandboxed in CI via the flake checks.
* `copilot/hooks` is a shell port of `claude/hooks` for Copilot CLI (camelCase payloads, JSON decisions on stdout instead of exit-2 blocks; see `copilot/hooks/mohan-hooks.json`); `pi/agent/extensions/hooks` is the TypeScript port. Keep the ports in sync when one changes.
* The same port rule applies to the prompt config: `copilot/copilot-instructions.md`, `copilot/agents/*.agent.md`, and `copilot/skills/*/SKILL.md` mirror `claude/CLAUDE.md`, `claude/agents/*.md`, `claude/skills/*/SKILL.md`, and `claude/commands/*.md` (adapted surfaces: Copilot has no custom slash commands, agent frontmatter uses `tools: [...]` lists, skills use YAML `name:` + `description:` frontmatter, and slash-command-equivalent skills like `/arch` and `/featurePlan` live at `claude/commands/<x>.md` while their Copilot counterparts live at `copilot/skills/<x>/SKILL.md`). A change to a charter, agent, command, or skill on one side must land on the other.
* Before writing a duplicate file across `claude/` and `copilot/`, check whether it belongs in `agents/` instead: if it's a script, template, or other artifact with zero tool-specific content (like `agents/skills/arch/`'s template and injection script), it goes in `agents/` once and both tools reference it by its fixed `~/.agents/...` path. Prose that merely reads similarly (charters, agent personas, skill instructions) still doesn't qualify, since Claude Code and Copilot CLI have no shared way to include one another's files; that keeps following the port rule above.
* Deployment exceptions, do not "fix" them: `~/.config/nvim` is an out-of-store symlink to this checkout (LazyVim writes `lazy-lock.json`), and `~/.claude/settings.json` is a writable copy (Claude Code writes permission grants into it).
* Secrets never enter this repo; machine-local values belong in `~/.zshrc.local` (sourced by `nix/zsh.nix`). On first adoption on a machine, `setup.sh`'s `migrate_pre_nix_dotfiles` folds any pre-existing `~/.zshrc`/`~/.zshenv` content into `~/.zshrc.local` before Home Manager takes the files over, so nothing is silently lost.
* Claude Code and the npm global CLIs are intentionally not pinned by Nix; they install natively via activation scripts only when missing.
